MENU API BACKLOG

How to use
- Each line is one API ticket.
- Keep it as [ ] until the endpoint is implemented and manually verified.
- Change [ ] to [x] only when both are done.
- These endpoints are not implemented yet in the current codebase.
- All ids are UUID values unless noted otherwise.
- Route style follows the existing project convention: top-level resource paths without an /api prefix.

Common auth and permission assumptions
- Protected endpoints require header: Authorization: Bearer <accessToken>
- Suggested permission codes to add before implementation:
  - MENUS_READ
  - MENUS_CREATE
  - MENUS_UPDATE
  - MENUS_DELETE

Common validation assumptions
- code and name are normalized and trimmed
- displayOrder must be greater than or equal to 0
- basePrice must be greater than or equal to 0
- minSelect and maxSelect must be greater than or equal to 0 when present
- minSelect must be less than or equal to maxSelect when both are present
- minSelectOverride must be less than or equal to maxSelectOverride when both are present
